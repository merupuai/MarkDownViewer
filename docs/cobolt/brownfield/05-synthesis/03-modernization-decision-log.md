# 20 — Modernization Decision Log

A precursor to ADRs in 28-modernization-architecture-decisions.md. Records the high-leverage decisions so far.

## DEC-001: Treat the rendering pipeline as the security trust boundary

The application has no network surface and no auth model. The realistic threats are content-driven (hostile markdown). DOMPurify + (eventually) CSP + path-containment in resolveImage become the trust boundary.

**Alternatives considered**:
- Accept a wider trust boundary (the OS file ACL) — rejected because a hostile .md file is in scope.
- Lower mermaid `securityLevel` to `strict` — rejected because users use HTML-in-mermaid features.

**Consequence**: future code that injects HTML must either pass through DOMPurify or be re-sanitized. SEC-001 / MOD-001 enforces this for mermaid SVG output.

## DEC-002: Forward-invest, don't replatform

Codebase is well-structured (~2000 LOC TS, no architectural debt). Replatform would be wasted effort. Modernization milestones layer on top of the existing two-process model.

**Alternatives considered**:
- Migrate to Tauri (Rust + WebView). Rejected because Electrobun's Bun runtime gives sub-process performance comparable to Tauri without forcing a Rust learning curve.
- Migrate to Electron. Rejected because Electron's bundle size (~150 MB) is materially worse than Electrobun's (~30 MB).

## DEC-003: Multi-format editor as a milestone, not a separate project

The in-flight multi-format editor plan (commit af327fe) builds on the existing rendering pipeline. Treating it as a milestone keeps the codebase coherent and avoids fork-and-merge churn.

## DEC-004: Don't ship docker-compose; remediate CIS upstream

The 4 CIS Docker findings (`OPS-001`) are on `_cobolt-docker/docker-compose.yml` — auto-generated by `/cobolt-init` for optional dev infra. The app does not author or distribute this file. Remediation belongs to the CoBolt template upstream, not in this repo.
