# Modernization Security Requirements — MarkDownViewer

## 1. Threat Model (STRIDE applied to a desktop content-viewer)

| STRIDE | Realistic in this app? |
|---|---|
| Spoofing | NO — single-user desktop app, no identity model |
| Tampering | YES — hostile markdown content (the input is attacker-controlled) |
| Repudiation | NO — no multi-user audit |
| Information Disclosure | YES — local-file exfil via image src (SEC-002), CSS-driven leaks (SEC-003) |
| Denial of Service | LOW — bounded by file caps and DOM size |
| Elevation of Privilege | NO — process runs as the user; no setuid; no helper |

The realistic adversary is a **hostile markdown file** opened by an unsuspecting user. Treat the file as untrusted input even though the OS user opened it deliberately.

## 2. Security Controls (mandatory)

### SR-01 (M1) — Mermaid SVG re-sanitization

After `mermaid.render()`, the resulting SVG MUST pass through `DOMPurify.sanitize` (with SVG profile) before being injected into the DOM. Closes SEC-001 / MOD-001.

### SR-02 (M1) — `resolveImage` path containment

`src/bun/index.ts::resolveImage` MUST reject any `src` whose `path.resolve(docDir, src)` does NOT begin with `docDir + path.sep` (or with one of a configured allowlist of subdirs). Closes SEC-002 / MOD-002.

### SR-03 (M1) — DOMPurify style attribute scoping

The DOMPurify config MUST use `ALLOWED_CSS` (or equivalent) to forbid `url()` and `@import` in inline `style` attributes. Closes SEC-003 / MOD-003.

### SR-04 (M1) — Renderer Content Security Policy

`src/mainview/index.html` MUST include a `<meta http-equiv="Content-Security-Policy">` tag with at minimum: `connect-src 'none'; frame-src 'none'; object-src 'none'; base-uri 'none'`. Inline scripts and styles are permitted (`'unsafe-inline'`) only because they are application-controlled assets. Closes SEC-007 / MOD-006.

### SR-05 (M1) — Image MIME enforcement

`resolveImage` MUST refuse to return a data URL for any extension not in the explicit allowlist (png/jpg/jpeg/gif/svg/webp/bmp/ico/avif). Today the function falls through to `application/octet-stream`; that fall-through MUST be removed. Closes part of SEC-002.

### SR-06 (M1) — Front-matter parse error visibility

Failures in `gray-matter::matter()` MUST surface as a `front-matter parse error` banner in the rendered output, with the underlying message. Closes SEC-004 / MOD-004.

### SR-07 (M2) — OSV / `bun audit` scan in CI

CI on every PR MUST run a vulnerability scan over `bun.lock` and fail the build on any HIGH or CRITICAL CVE. Closes SCA-002 / MOD-009.

### SR-08 (M2) — Code signing

Distribution binaries MUST be code-signed (Authenticode for Windows, Apple Developer ID for macOS) and (mac) notarized + stapled. Closes ENH-020 / ENH-021.

### SR-09 (M2) — Hostile-content fixture corpus

The Playwright test suite MUST include hostile fixtures for: hostile mermaid, hostile image src (path traversal), hostile CSS (url, @import), malformed front-matter. Each fixture MUST verify that no HTML / file content is leaked into the rendered DOM. Closes part of FR-12.

## 5. Defense-in-Depth Layers

```
Layer 1: markdown-it parsing (html: true)
   ↓ (rendered HTML, possibly hostile)
Layer 2: DOMPurify with hardened allowlist (SR-03)  ← MUST gate all HTML
   ↓ (sanitized HTML)
Layer 3: Renderer DOM (under CSP — SR-04)
   ↓ (SVG injection from mermaid)
Layer 4: Mermaid SVG re-sanitize (SR-01)            ← MUST gate mermaid
   ↓ (sanitized SVG)
Layer 5: Image data URL resolver (SR-02 + SR-05)    ← MUST gate file reads
   ↓ (data URL, contained to docDir, allowed MIMEs only)
Layer 6: OS file ACL (out of app's control)
```
